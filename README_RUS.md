# Remarkable News

Надёжный self-hosted сервис для сборки RSS в PDF и доставки выпусков на reMarkable.

Это доработанный и активно поддерживаемый форк проекта [ktibr0/Remarkable_morning_news](https://github.com/ktibr0/Remarkable_morning_news), который сейчас архивирован. Оригинальный проект был построен вокруг workflow n8n; эта версия превращает идею в отдельное веб-приложение с постоянным хранилищем, повторами, архивом PDF, встраиванием картинок и прямой отправкой в reMarkable Cloud через [ddvk/rmapi](https://github.com/ddvk/rmapi/).

Веб-интерфейс по умолчанию открывается на английском, русский язык можно выбрать в правом верхнем углу.

Конвейер:

```text
RSS -> SQLite/очередь -> извлечение и проверка статей -> HTML -> Gotenberg -> архив PDF -> rmapi
```

Ошибка одной ленты, статьи или картинки не отменяет весь запуск. Неудачные URL записываются в журнал, повторяются с увеличивающейся задержкой и могут быть вручную возвращены в очередь. PDF строится только из статей, прошедших проверки.

## Возможности

- Веб-панель для RSS-подписок, расписания, оформления PDF, сохранённых выпусков, журнала и доставки на reMarkable.
- Английский и русский интерфейс.
- ETag и Last-Modified для экономного чтения RSS.
- Мягкая обработка частично повреждённых RSS-лент.
- Извлечение полного текста через Trafilatura, если RSS содержит только анонс.
- Проверка минимальной длины полезного текста и статусы статей: `pending`, `failed`, `dead`, `ready`.
- Экспоненциальные повторы и ручной повтор неудачных URL.
- Журнал с точным URL, источником и причиной ошибки.
- Поиск главной картинки, локальный кэш изображений и встраивание картинок в PDF.
- Архив PDF с настраиваемым сроком хранения.
- Ручная повторная отправка любого сохранённого выпуска.
- Опциональная автоматическая отправка в reMarkable Cloud после плановых и ручных сборок.
- SQLite в WAL-режиме, постоянная папка `./data` рядом с проектом и непривилегированный пользователь приложения.
- Multi-stage Docker-образ с закреплёнными зависимостями и закреплённой сборкой rmapi.
- Облегчённый сервис `gotenberg/gotenberg:8-chromium` без LibreOffice.

## Быстрый запуск

Требуется:

- Docker Engine с Compose v2.
- Доступ сервера в интернет.

```bash
docker compose build
docker compose up -d
```

Откройте веб-интерфейс:

```text
http://<server-ip>:8000
```

Добавьте RSS на вкладке Feeds, настройте сбор и PDF на вкладке Layout & collection, затем нажмите Build edition.

## Подключение reMarkable

1. Откройте вкладку reMarkable.
2. Перейдите по ссылке получения одноразового кода.
3. Вставьте код в форму и нажмите Connect.
4. Приложение проверит подключение реальным запросом к облаку.
5. Укажите папку, например `/News`.
6. Включите Automatically send editions, если хотите автоматически отправлять каждый готовый выпуск в reMarkable Cloud.

Если в volume остались старые или повреждённые токены rmapi, нажмите Reset tokens и авторизуйтесь заново со свежим одноразовым кодом.

Учётные данные rmapi хранятся в Docker-volume `remarkable-news_rmapi_config`. Они не лежат в SQLite, не копируются в образ и никогда не должны попадать в Git.

## Хранение и резервные копии

Рабочие файлы специально вынесены из образа:

- `./data`: SQLite, PDF, кэш картинок и журналы (`news.db`, `editions/`, `images/`, `logs/`).
- `remarkable-news_rmapi_config`: токены reMarkable Cloud и локальная конфигурация rmapi.

Пример резервной копии:

```bash
tar czf remarkable-news-data.tgz -C data .
docker run --rm -v remarkable-news_rmapi_config:/source:ro -v "$PWD":/backup alpine \
  tar czf /backup/remarkable-news-rmapi.tgz -C /source .
```

Для согласованной копии SQLite лучше остановить приложение перед копированием и запустить его после:

```bash
docker compose stop app
docker compose start app
```

## Рекомендуемые настройки

- Минимум полезных символов: 300-600. Чем выше значение, тем меньше пустых страниц, но короткие новости могут быть пропущены.
- Количество попыток: 5.
- Первая повторная попытка: через 15 минут.
- Look back: 48 часов, чтобы первый запуск не забрал слишком старый архив.
- Картинки: 3 МБ на изображение. Слишком большие или недоступные картинки пропускаются.
- Хранение: 30 дней. Старые PDF и закрытые старые ошибки удаляются; записи статей остаются для дедупликации.
- Каждая статья с новой страницы: удобно для навигации на планшете, но PDF получается длиннее.

## Эксплуатация

```bash
docker compose logs -f app
docker compose ps
docker compose pull gotenberg
docker compose build --pull app
docker compose up -d
```

API-документация доступна на `/docs`.

Сервис рассчитан на один планировщик и один процесс приложения. Не увеличивайте количество workers без переноса блокировок и владения очередью из процесса приложения.

## Безопасность

В панели нет встроенной учётной записи. Она рассчитана на доверенную домашнюю сеть.

Не публикуйте порт `8000` напрямую в интернет. Для удалённого доступа используйте VPN или reverse proxy с TLS и авторизацией. Gotenberg доступен только внутри Docker-сети.

Не коммитьте эти файлы и папки:

- `.githubtoken`
- `.ssh_creds/`
- `.venv/`
- `data/`
- `config/`
- конфигурацию и токены rmapi
- сгенерированные PDF и резервные копии

В проектном `.gitignore` и `.dockerignore` эти пути уже исключены.

## Диагностика

Типичные ошибки:

- `feed_error`: лента не загрузилась или не содержит читаемых записей.
- `feed_warning`: feedparser нашёл нарушение формата RSS, но записи удалось прочитать.
- `article_error`: полный текст не получен, а RSS-текста недостаточно.
- `pdf_error`: Gotenberg не создал PDF.
- `upload_error`: rmapi не отправил PDF.

Полный аварийный вывод rmapi пишется в `/data/logs/rmapi.log` с ротацией. Веб-журнал показывает только практическую причину ошибки.

Открытые проблемы видны на вкладке Log. Retry failed URLs возвращает в очередь как временные ошибки, так и статьи с исчерпанным лимитом попыток. Ошибки RSS-лент повторяются автоматически при следующем плановом запуске.

## Разработка

```bash
uv sync --extra dev
uv run ruff check .
uv run pytest
```

Зависимости закреплены в `uv.lock`. Docker BuildKit по возможности переиспользует слои uv и Go.

## Переход со старого workflow n8n

Старый файл `Remarkable_news.json` оставлен как справочный.

Рекомендуемый порядок перехода:

1. Запустите этот Compose-стек на другом порту либо остановите старые контейнеры `rmapi_api` и Gotenberg.
2. Добавьте RSS-ленты в новой панели.
3. Установите интервал, лимиты записей, lookback и папку доставки как в старом workflow.
4. Авторизуйте rmapi.
5. Выполните ручную тестовую сборку.
6. Проверьте PDF и доставку.
7. Отключите старое расписание n8n.
